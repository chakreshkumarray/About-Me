First, I noticed a major flaw in how the database transaction is handled. The code commits the new product to the database immediately, before it even tries to create the inventory record. If the inventory creation fails for any reason—like a missing field or a database hiccup—I'm going to end up with an "orphaned" product in the system that has no associated warehouse or stock.

Second, there is a fundamental issue with the business logic and data modeling. The code takes the warehouse_id and attaches it directly to the Product object. The requirements explicitly state that a product can exist in multiple warehouses, but SKUs must remain unique. If I lock a product to a single warehouse ID, the only way I could add that same item to a second warehouse later would be to create a duplicate product record. That would violate the unique SKU rule and create a mess in the database.

Third, there are no safety nets. The code blindly assumes the incoming JSON payload is perfect. If a client forgets to send an initial_quantity or sends a duplicate SKU, the server is just going to crash and throw a generic 500 Internal Server Error, which is a bad user experience.

Finally, the endpoint returns a default 200 OK status. Since I am creating a new resource in the database, it really should be returning a 201 Created status to follow REST API best practices.

My Logic & Solution
To fix this, I completely restructured how the endpoint processes the data.

To solve the database transaction issue, I made sure both operations happen together atomically. Instead of committing the product immediately, I push it to the database just enough to generate the new Product ID, but I hold off on the final commit. I then use that new ID to build the inventory record. Only after both the product and the inventory are successfully staged do I commit them together. If anything fails along the way, I roll the whole thing back so the database stays clean.

To fix the business logic flaw, I removed the warehouse_id from the product creation step entirely. A product shouldn't care where it's stored. Instead, I only attach the warehouse_id to the new Inventory record. This perfectly aligns with the requirements: I now have one central product record (with a unique SKU), and I can tie it to as many different warehouse inventory records as I need to.

For the safety nets, I added strict input validation at the very beginning of the request. I check if all the required fields are there, and I make sure the price is safely converted to a decimal. I also wrapped the database operations in a try-catch block. Now, if someone submits a duplicate SKU, instead of crashing the server, I catch that specific error and return a polite 409 Conflict message telling the client exactly what went wrong.

Lastly, I updated the return response to explicitly send back a 201 Created status code along with the new product's ID.
